# Career RAG Load Validation Report

**Date:** 2026-07-14

**Branch:** `codex/week3-reoptimization`

## Scope and Safety

This report covers the local P0/P1-mechanism prototype with deterministic database, LLM, and embedding substitutes. Tests did not read or print `.env`, did not call DeepSeek or DashScope, and used only synthetic session, run, job, case, and evidence identifiers.

The measurements are a local engineering baseline. They demonstrate state isolation, async overlap, recovery contracts, and bounded provider concurrency; they are not internet-scale capacity claims and do not predict real provider latency.

## Complex Business Scenarios

The focused matrix executed 48 tests in 1.64 seconds:

- Targeted Intent Agent consultation with a named role or company.
- Explore consultation with at most three directions and no redundant second Agent call after selection.
- One bounded clarification followed by locked Match Brief fields.
- Canonical `plan_hash`, stale-plan 409 response, and structured recovery guidance.
- Result projection requiring JD evidence and rejecting hidden or hard-constraint-failing jobs.
- Public explanation with bounded case evidence and no raw state, prompt, provider error, or full resume.
- Dual-space cold start, confident reranking, implicit failure degradation, and cancellation propagation.
- Concurrent run snapshots, result isolation, status polling, and competing execute behavior.

Result: **48 passed, 0 failed**. One pre-existing Starlette/httpx deprecation warning was reported.

## Retrieval Parallelism

The retrieval tests use event rendezvous points as deterministic evidence of overlap, plus a wall-clock guard:

- SQL/metadata hard filtering completes first and produces the allow-list.
- BM25 and dense branches then each hold for 100 ms and must finish together in under 180 ms.
- Explicit job retrieval and implicit case retrieval each hold for 100 ms and must finish together in under 180 ms.
- Implicit ordinary exceptions degrade to explicit results.
- Explicit retrieval failures remain run failures.
- `asyncio.CancelledError` propagates instead of being hidden by degradation.

The focused retrieval/provider suite result was **21 passed, 0 failed**.

## Defects Found and Resolved

- The new root route originally rendered the upload workbench and invoked capabilities. The isolated `/` and `/workspace` route test failed first, then passed after route separation.
- The introduction initially focused its heading on first load, producing an oversized focus frame. A regression test now requires no initial focus while preserving focus transfer after step changes.
- On Windows, Playwright-owned Vite shutdown can delay command completion. Verification now reuses an explicitly managed test server; this changes test process management only, not application behavior.

## Remaining Production Limitations

- The load suite does not measure a real PostgreSQL/pgvector server under network and disk contention.
- It does not consume real DeepSeek or Qwen quotas, so provider latency and rate-limit recovery require a separately authorized staging test.
- Execution durability remains `process_local`; a process restart can interrupt background execution even though snapshots are stored by run ID.
- Measurements are from one Windows development host, not a multi-instance deployment behind a load balancer.
- These tests validate mechanism correctness and local async behavior, not recommendation quality on a labelled career-matching dataset.
